# Deep Dive: Monitor Real-time System Health (/dev/monitor)

เอกสารสรุปฟังก์ชันการทำงานฝั่ง Backend และ Logic ของหน้า Monitor Dashboard สำหรับนำไปใช้ประกอบการทำสไลด์

---

## 📊 1. ส่วนสรุปภาพรวม (Overall Summary Cards)
ส่วนที่แสดง Card ด้านบนสุดของหน้าจอ เพื่อให้เห็นสุขภาพระบบในพริบตา

| ชื่อฟังก์ชัน | Path ไฟล์ | รายละเอียดการทำงานและวัตถุประสงค์ |
| :--- | :--- | :--- |
| `usePerformanceMetrics` | `src/hooks/useAdminMonitor.tsx` | **การทำงาน:** ดึงข้อมูลจากตาราง `server` มาคำนวณค่าเฉลี่ย (Aggregation) ของ CPU, Memory และ Disk Usage<br>**วัตถุประสงค์:** เพื่อแสดง "สรุปสถิติทรัพยากร" ของเซิร์ฟเวอร์ทั้งหมดที่สถานะเป็นปกติ (Healthy) ในหน้าเดียว |
| `useErrorLogStats` | `src/hooks/useAdminMonitor.tsx` | **การทำงาน:** ดึงรายการ Log จากตาราง `error_logs` ล่าสุด 500 รายการมานับจำนวนแยกตาม Level<br>**วัตถุประสงค์:** เพื่อแสดง "จำนวน Error รวม" ในรอบ 24 ชั่วโมงล่าสุด ช่วยให้ประเมินความรุนแรงของปัญหาได้ทันที |

---

## 🖥️ 2. Tab: Servers (โครงสร้างพื้นฐาน)
ส่วนตรวจสอบสถานะและทรัพยากรของเซิร์ฟเวอร์แยกรายเครื่อง

| ชื่อฟังก์ชัน | Path ไฟล์ | รายละเอียดการทำงานและวัตถุประสงค์ |
| :--- | :--- | :--- |
| `useServerHealth` | `src/hooks/useAdminMonitor.tsx` | **การทำงาน:** ดึงข้อมูลจากตาราง `public.server` และเชื่อมต่อระบบแจ้งเตือนการเปลี่ยนแปลงแบบสดๆ (Live Update)<br>**วัตถุประสงค์:** เพื่อให้หน้าจอ "อัปเดตข้อมูลเซิร์ฟเวอร์อัตโนมัติ" ทันทีที่มีการเปลี่ยนแปลง โดยไม่ต้องกด Refresh หน้าเว็บ |
| `computeServerStatus` | `src/hooks/useAdminMonitor.tsx` | **การทำงาน:** รับข้อมูล CPU/Mem/Disk แล้วคืนค่าสถานะเป็น `healthy`, `warning` (>80%), หรือ `critical` (>90%)<br>**วัตถุประสงค์:** เป็นจุดศูนย์กลางของ "ตรรกะการตัดสินใจ" ว่าเซิร์ฟเวอร์เครื่องไหนกำลังประสบปัญหาตามเกณฑ์ที่กำหนด |

---

## 🧪 3. Tab: Data Pipelines (ท่อส่งข้อมูล)
ส่วนตรวจสอบการรันกระบวนการข้อมูล (Data Processing)

| ชื่อฟังก์ชัน | Path ไฟล์ | รายละเอียดการทำงานและวัตถุประสงค์ |
| :--- | :--- | :--- |
| `useDataPipelines` | `src/hooks/useAdminMonitor.tsx` | **การทำงาน:** ดึงข้อมูลจากตาราง `public.data_pipeline` เพื่อตรวจสอบสถานะการทำงานล่าสุด<br>**วัตถุประสงค์:** เพื่อดูว่าระบบประมวลผลข้อมูลทำงานปกติหรือไม่ โดยแบ่งสถานะหลักเป็น **Active** (ปกติ), **Warning** (ควรระวัง), และ **Failed** (ล้มเหลว) |

---

## ⚠️ 4. Tab: Errors (การวิเคราะห์ความผิดพลาด)
ส่วนวิเคราะห์เชิงสถิติของ Error ที่เกิดขึ้นในระบบ

| ชื่อฟังก์ชัน | Path ไฟล์ | รายละเอียดการทำงานและวัตถุประสงค์ |
| :--- | :--- | :--- |
| `useErrorLogStats` | `src/hooks/useAdminMonitor.tsx` | **การทำงาน:** นอกจากจะดึงข้อมูลแล้ว ในหน้านี้จะมีการทำ Logic เพิ่มเติมเพื่อหา `topIssues` (Grouping Message)<br>**วัตถุประสงค์:** เพื่อระบุ "ปัญหาที่พบบ่อยที่สุด" (Top 5 Issues) และแสดงกราฟแนวโน้ม (Trend) เพื่อดูว่า Peak ของ Error เกิดขึ้นในช่วงเวลาใด |

---

## 🌐 5. Tab: External APIs (การเชื่อมต่อภายนอก)
ส่วนตรวจสอบสุขภาพของ API ภายนอกที่ระบบเชื่อมต่อด้วย (เช่น Facebook, Line, Google)

| ชื่อฟังก์ชัน | Path ไฟล์ | รายละเอียดการทำงานและวัตถุประสงค์ |
| :--- | :--- | :--- |
| `useExternalAPIStatus` | `src/hooks/useAdminMonitor.tsx` | **การทำงาน:** ตรวจสอบสถานะการเชื่อมต่อกับบริการภายนอก (เช่น Facebook, Google) ตลอดเวลา<br>**วัตถุประสงค์:** เพื่อคอยเฝ้าระวังว่าบริการข้างนอก "ล่ม" หรือ "ทำงานช้า" จนส่งผลกระทบต่อระบบของเราหรือไม่ |

---

> [!TIP]
> **Real-time Feature:** ทุกหน้าในหน้านี้ใช้ `Supabase Realtime` ผ่านฟังก์ชัน `supabase.channel()` เพื่อทำ Postgres Changes Subscription ทำให้ข้อมูลบนหน้าจออัปเดตเองโดยไม่ต้อง Refresh หน้าเว็บเมื่อข้อมูลในฐานข้อมูลเปลี่ยน (Low Latency Updates)
